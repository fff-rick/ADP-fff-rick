# 《基于 AI Agent 的自动化运维调度系统设计与实现》第二版项目策划书

## 一、项目名称

《基于 AI Agent 的自动化运维调度系统设计与实现》

英文名称：

> AI-Agent-Assisted Automated Operation and Maintenance Scheduling System

------

## 二、第二版修订说明

相较第一版，第二版重点做了以下调整：

- 收缩项目范围，聚焦“AI 辅助运维调度”，不将“完全自治运维”作为首版交付目标
- 明确系统边界，区分 AI 负责的分析决策层与调度系统负责的执行控制层
- 增加安全约束，包括命令白名单、人工确认、权限隔离、审计日志
- 降低技术复杂度，弱化非核心组件，优先保证主链路可实现
- 重写实施计划与性能指标，使其更符合毕业设计和原型系统的落地节奏

------

## 三、项目背景

随着云原生、微服务以及容器化技术的发展，现代运维场景普遍存在以下问题：

- 服务实例数量多，人工巡检成本高
- 故障类型复杂，定位过程依赖经验
- 告警与日志信息量大，处理效率低
- 传统调度平台偏重定时执行，缺乏智能分析能力

传统运维系统通常能够完成：

- 定时任务执行
- 固定脚本下发
- 简单流程编排

但在以下方面仍然不足：

- 自然语言理解能力不足
- 故障诊断自动化程度低
- 缺少任务规划与执行反馈闭环

近年来，大语言模型与 Agent 技术的发展，使系统具备以下能力：

- 将自然语言转化为结构化任务
- 基于上下文生成诊断步骤
- 对日志和执行结果做语义分析
- 辅助生成修复建议

因此，设计一套“基于 AI Agent 的自动化运维调度系统”，具有较强的研究意义与实践价值。

## 四、项目定位与边界

### 4.1 项目定位

本项目定位为：

> 一套面向典型运维场景的 AI 辅助调度原型系统。

系统强调：

- AI 辅助理解与分析
- 调度系统稳定执行
- 人机协同决策
- 可追踪、可回放、可审计

### 4.2 首版范围

第二版将首版能力聚焦于以下三类典型场景：

- 周期性数据库备份任务
- Web 服务可用性诊断任务
- Redis 常见性能问题诊断任务

### 4.3 非目标范围

以下能力不作为首版必须完成项：

- 面向任意运维任务的通用自治执行
- 无限制自由命令生成与执行
- 全量云原生集群治理
- 大规模生产级多租户平台能力
- 高风险故障的全自动修复

------

## 五、项目目标

本项目旨在设计并实现一套结合 AI Agent 与分布式调度能力的自动化运维原型平台，使系统能够：

- 接收自然语言形式的运维目标或故障描述
- 将输入解析为结构化任务或诊断请求
- 基于预定义工具与模板生成执行计划
- 将任务分发至 Worker 节点执行
- 采集执行结果、日志与状态信息
- 利用 AI 对结果进行分析并给出诊断结论
- 在低风险场景下生成修复建议，并支持人工确认后执行

最终形成如下闭环：

```text
输入 -> 解析 -> 规划 -> 调度 -> 执行 -> 分析 -> 反馈
```

需要特别说明的是，第二版目标强调“AI 辅助决策闭环”，而不是“完全无人值守自治闭环”。

## 六、系统功能需求

### 6.1 用户与权限管理模块

功能包括：

- 用户登录认证
- JWT 身份鉴权
- 角色权限控制
- 操作审计日志

建议角色划分：

- 管理员：维护系统配置、Worker 和策略
- 运维用户：提交任务、查看执行结果、确认修复操作
- 只读用户：查看任务与日志

### 6.2 AI 目标解析模块

用户输入自然语言任务，例如：

```text
每天凌晨备份 MySQL 数据库
```

系统解析为结构化结果，例如：

```json
{
  "intent": "create_scheduled_backup",
  "target_type": "mysql",
  "schedule": "0 0 * * *",
  "risk_level": "low"
}
```

模块职责：

- 自然语言意图识别
- 参数抽取
- 输出 JSON 结构化结果
- 对无法识别或高风险输入进行拦截

### 6.3 AI 任务规划模块

该模块是系统的核心智能模块，但需要建立在“受控工具集”之上。

例如用户输入：

```text
Redis 响应速度过慢
```

系统输出的不是任意命令，而是受控步骤，例如：

```text
1. 检查 Redis 基本存活状态
2. 获取内存使用信息
3. 查询 slowlog
4. 查看当前连接数
5. 分析最近一段时间 CPU 指标
```

模块职责：

- 生成诊断步骤
- 为步骤绑定可执行工具或命令模板
- 为每一步设置前置条件、超时和预期输出
- 标记风险级别和是否需要人工确认

### 6.4 分布式任务调度模块

系统支持多 Worker 节点协同执行，核心能力包括：

- Worker 注册与心跳上报
- 任务入队与分发
- 任务状态流转
- 超时控制
- 失败重试
- 执行结果回传

首版调度目标不是追求极致吞吐，而是保证：

- 正确性
- 可观测性
- 故障可恢复性

### 6.5 Worker 执行模块

Worker 负责在受控环境中执行任务，首版建议支持以下类型：

| 类型 | 功能 |
| --- | --- |
| Shell Worker | 执行本机受控命令 |
| SSH Worker | 执行远程主机受控命令 |
| HTTP Worker | 调用外部 API 或健康检查接口 |

`Docker Worker` 可以作为扩展项，不建议与首版核心链路同时强绑定。

### 6.6 执行结果分析模块

系统对以下信息进行分析：

- 标准输出与标准错误输出
- 健康检查结果
- 服务返回码
- 关键日志片段

AI 输出内容包括：

- 故障类型推测
- 可能原因排序
- 建议下一步动作
- 是否满足修复条件

### 6.7 修复建议与人工确认模块

第二版不默认执行高风险修复命令。

系统应支持：

- 自动生成修复建议
- 低风险动作自动执行
- 中高风险动作等待人工确认
- 执行后进行结果验证

低风险动作示例：

- 重试健康检查
- 重新拉取只读状态信息
- 清理临时文件

中高风险动作示例：

- 重启服务
- 修改配置
- 删除数据
- 批量远程操作

### 6.8 经验库与知识检索模块

系统记录以下信息：

- 历史故障案例
- 执行计划
- 任务结果
- 修复建议与最终处理结果

首版目标是构建“结构化经验库”，用于：

- 历史案例检索
- 相似问题辅助诊断
- 提升 AI 提示上下文质量

首版不强依赖独立向量数据库，可先以关系型存储加文本检索为主，后续再扩展 RAG。

## 七、系统总体架构

整体架构如下：

```text
+--------------------+
|       Web UI       |
+---------+----------+
          |
          v
+--------------------+       +----------------------+
|   API Gateway      |-----> |   Auth / Audit       |
+---------+----------+       +----------------------+
          |
          v
+---------------------------------------------------+
|                Control Plane                       |
|---------------------------------------------------|
| Task Parser | Task Planner | Policy Engine        |
| Scheduler   | Result Analyzer | Case Memory       |
+---------------------------------------------------+
      |                 |                  |
      v                 v                  v
+-----------+     +-----------+      +-----------+
|   MySQL   |     |   Redis   |      |  LLM API  |
+-----------+     +-----------+      +-----------+
      |
      v
+------------------------------+
|        Worker Cluster        |
|------------------------------|
| Shell Worker / SSH / HTTP    |
+------------------------------+
```

架构设计原则：

- AI 负责“理解、规划、分析、建议”
- Scheduler 负责“派发、控制、重试、追踪”
- Worker 负责“受控执行，不做自主决策”
- Policy Engine 负责“风控、审批、白名单、拦截”

## 八、核心流程设计

### 8.1 任务创建流程

```text
1. 用户提交自然语言任务
2. AI 解析为结构化任务
3. Policy Engine 校验任务是否合法
4. Scheduler 创建任务实例
5. 任务下发至 Worker 执行
6. 结果回传并持久化
7. 系统展示执行结果
```

### 8.2 故障诊断流程

```text
1. 用户输入故障现象
2. AI 生成诊断计划
3. Planner 将计划映射到受控工具
4. Scheduler 串行或并行下发任务
5. Worker 返回观测结果
6. AI 汇总分析并输出诊断结论
7. 系统给出修复建议或下一步计划
```

### 8.3 修复执行流程

```text
1. AI 生成修复建议
2. Policy Engine 根据风险等级判定
3. 低风险动作直接执行
4. 中高风险动作进入人工确认
5. 执行完成后自动校验结果
6. 全量记录审计日志
```

## 九、调度与执行设计

### 9.1 任务状态机

建议任务状态至少包括：

- `pending`
- `queued`
- `running`
- `success`
- `failed`
- `timeout`
- `cancelled`
- `waiting_approval`

### 9.2 调度策略

首版建议采用简单且可靠的调度方式：

- 基于 Worker 类型匹配任务
- 基于 Worker 在线状态过滤
- 基于当前负载做基础均衡
- 支持失败重试和超时回收

不建议在首版引入过于复杂的调度算法。

### 9.3 执行约束

为确保安全与稳定，执行层必须具备：

- 命令模板化
- 参数校验
- 超时限制
- 输出长度限制
- 幂等性约束
- 全链路审计

### 9.4 人工确认机制

以下场景默认需要确认：

- 涉及服务重启
- 涉及配置修改
- 涉及跨主机批量操作
- 涉及删除或覆盖数据

## 十、安全设计

第二版必须补足安全部分，否则方案不具备实际合理性。

### 10.1 权限控制

- 用户权限与任务权限隔离
- Worker 使用最小权限原则
- 敏感操作要求更高角色权限

### 10.2 命令白名单

系统不直接执行 LLM 任意生成的自由命令。

首版建议采用：

- 工具白名单
- 命令模板白名单
- 参数范围校验

### 10.3 审计与追踪

系统必须记录：

- 谁发起了任务
- AI 生成了什么计划
- 最终执行了什么动作
- 哪个 Worker 执行
- 结果是什么

### 10.4 风险分级

建议定义三类风险：

- 低风险：查询类、只读类、重复执行影响小
- 中风险：服务级变更、单机重启
- 高风险：删除数据、批量变更、配置覆盖

## 十三、系统创新点

第二版建议将创新点写得更具体、更可验证。

### 创新点 1：自然语言驱动的受控运维任务生成

不是简单“让 AI 生成命令”，而是：

- 先做意图识别
- 再映射到模板化工具
- 最终生成结构化执行计划

创新点在于“自然语言到受控执行”的桥接机制。

### 创新点 2：AI 规划与调度系统解耦

AI 负责规划，调度器负责执行控制，两者职责明确：

- 降低 AI 直接控制执行带来的不确定性
- 让系统具备更强可解释性和可审计性

### 创新点 3：基于风险分级的人机协同修复机制

不是一味追求全自动恢复，而是：

- 低风险自动化
- 中高风险人工确认
- 修复后自动校验

这更符合真实运维场景。

### 创新点 4：面向典型运维场景的案例复用机制

将历史任务、日志、诊断结论与修复方式沉淀为案例库，为后续相似问题提供辅助支持。

## 十五、性能与验收指标

第二版指标以“原型系统合理可证”为原则，不盲目追求生产级指标。

建议验收指标如下：

| 指标 | 目标 |
| --- | --- |
| Worker 节点数 | 10 到 30 |
| 并发任务数 | 100 到 500 |
| 普通任务调度延迟 | 小于 500ms |
| 单次 AI 解析或分析响应 | 2 到 8s |
| 任务执行成功率 | 大于 95% |
| 典型场景解析准确率 | 大于 85% |
| 修复建议可采纳率 | 大于 70% |

说明：

- 如果使用本地模型，AI 响应时间可以适当放宽
- 如果后续压测表现优于目标，可作为论文中的扩展成果

## 十六、测试与验证方案

为了增强方案说服力，第二版建议补充实验与验证设计。

### 16.1 功能验证

验证以下链路是否完整：

- 自然语言创建备份任务
- Nginx 故障诊断与建议输出
- Redis 性能诊断步骤生成
- 人工确认后执行低频修复动作

### 16.2 性能验证

验证以下指标：

- 多 Worker 并发下发能力
- 心跳与任务回传稳定性
- Redis 队列与 MySQL 持久化性能

### 16.3 效果验证

对比两种方式：

- 人工直接编写命令执行
- AI 辅助生成计划后执行

可比较指标：

- 任务创建耗时

## 十九、项目最终目标总结

本项目第二版的最终目标是：

> 构建一套融合 AI 理解能力与分布式调度能力的运维原型系统，在典型场景下实现“可控、可解释、可审计”的智能辅助运维闭环。

系统最终实现的不是“无限制自治”，而是：

```text
AI 理解任务
-> 生成受控计划
-> 调度系统稳定执行
-> AI 分析执行结果
-> 人机协同完成修复
```

这一定义更符合工程现实，也更适合作为毕业设计与论文实现目标。
